Project Statement

What this project is and why it was built

This project presents an interpretable air-quality assessment system that predicts AQI from pollutant readings while also explaining the strongest drivers behind each prediction. It was built to move beyond black-box forecasting and toward a form of environmental intelligence that is easier to justify, communicate, and evaluate.

The application combines prediction, explainability, comparative literature positioning, and validation evidence in one interface. The goal is not only to estimate AQI severity, but also to show why the model behaves as it does and where its methodology is stronger or more limited than prior work.

Prediction

Regression-first AQI assessment

The system predicts continuous AQI values and then maps them to severity buckets, preserving more detail than classification-only approaches.

Interpretability

Per-prediction explanations

SHAP values identify which pollutants and temporal features push AQI up or down for each individual case.

Validation

Cross-temporal reality check

Model behavior is compared against independent CPCB 2024 evidence to test whether learned patterns still hold outside the training window.

System Story

Why this system exists

Most air-quality models stop at the prediction itself. This project pushes further by treating explainability as a first-class requirement. Instead of returning a number with no context, the system connects the AQI result to its strongest pollutant drivers and temporal signals.

The design is meant for academic demonstration and decision-support storytelling. That is why the app includes a predictor, model insights, validation evidence, and this project narrative in one place.

Methodology

Pipeline from raw records to interpretable output

A document-style overview of the main stages used to build and justify the system.

01

Data audit

CPCB-derived records were screened for authenticity, missingness, and anomalies before modeling decisions were made.

02

Quality control

Null handling was tiered by severity, temporal outliers were examined, and risky data patterns were filtered rather than ignored.

03

Feature engineering

Cyclical seasonal encoding, lag variables, rolling statistics, and city-level context were introduced to capture temporal persistence.

04

Model comparison

Baselines and multiple regressors were tested using chronological evaluation, with XGBoost selected on overall generalization quality.

05

Explanation and validation

SHAP outputs were used for feature-level explanations and then checked against external 2024 evidence for consistency.

Literature Review

How the project compares with related work

Switch between comparison lenses to see where the present study is stronger and where prior work still has an advantage.

Methodological rigor

Chronological splitting, baseline models, and explicit data audit practices make the reported performance more believable.

Explainability

SHAP is used both globally and locally, giving the system a real explanation layer absent in the reviewed papers.

Validation depth

Independent 2024 CPCB evidence is used to test whether learned patterns generalize across time and source.

Scale

The project spans many cities rather than staying within a single city or an unspecified regional slice.

Meteorological features

Paper 2 includes temperature, pressure, humidity, and wind speed, which remain a major future-improvement area for this project.

Health outcomes

Paper 1 links AQI patterns to health indicators more directly, while this system uses AQI as a proxy for severity communication.

Station-level granularity

Some prior work operates closer to monitoring-station detail, whereas this project stays at city-level resolution.

Real-time ingestion

The current application demonstrates interpretable prediction well, but it is not yet a live streaming AQI platform.

Limitations

Known constraints

  • Training data ends in 2020, so post-2020 structural shifts matter.
  • Meteorological drivers are not yet included in the feature space.
  • Predictions are city-level, not station-level micro-assessments.
  • Supabase logging supports app usage tracking, not real-time ingestion.
Next Steps

Where the system can grow

  • Add meteorological and satellite-derived context variables.
  • Extend validation with more recent and more granular data.
  • Introduce station-aware modeling for within-city comparisons.
  • Expand the app toward live feeds and policy-facing reporting.
